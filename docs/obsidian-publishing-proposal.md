# Obsidian → Blog 自动发布方案（v1）

## 目标

为 `Dingo's Den` 建立一条尽可能低维护的内容管线：

- 自动从 Obsidian 笔记来源中发现候选内容
- 自动过滤明显不适合公开发布的内容
- 自动提炼成适合 blog 的文章草稿或正式文章
- 自动提交到当前 blog 仓库
- 通过 Vercel 自动完成线上部署

这套方案的重点不是“导入所有笔记”，而是：

- 让高质量笔记更容易进入 blog
- 把误发敏感内容的风险压到最低
- 把人工操作压缩到“例外情况处理”而不是“每篇都手工发”

## 用户诉求拆解

当前诉求有两个核心：

1. 尽量自动化维护 blog，减少手工复制、整理、发布
2. 需要严格防止隐私、未来计划、公司敏感信息等内容误发

因此这不是单纯的“同步系统”，而是一个：

- 内容发现系统
- 风险过滤系统
- 内容转写系统
- 自动发布系统

## 数据源选择

### 结论

默认主数据源选 **Obsidian 的 GitHub repo**。

### 为什么不优先选本地 vault

本地 vault 适合：

- 调试规则
- 本地预览
- 临时试跑

但不适合长期自动发布主链路，因为：

- 依赖单台机器在线
- 调度不稳定
- 不利于持续化运行
- 自动化平台难以直接安全访问你的本地文件系统

### 为什么不优先选 OSS

OSS 更适合做同步存储或备份，不适合做主内容源，因为：

- Markdown 结构和目录语义不一定稳定
- 版本历史不如 Git 清晰
- 不容易只处理增量变更
- 审核和回滚体验弱

### 为什么优先选 GitHub repo

GitHub repo 作为内容源的优点：

- 天然支持增量扫描
- 有 commit 历史，便于追踪变化
- 适合挂 GitHub Actions 定时运行
- 容易做白名单目录、黑名单目录、变更检测
- 适合未来做 PR 审核或回滚

## 总体架构

推荐采用“双仓库自动化”结构：

- 内容源仓库：Obsidian vault 的 GitHub repo
- 发布目标仓库：当前 blog repo

自动化任务运行在 blog repo 中，定期完成以下动作：

1. 拉取或读取 Obsidian repo 指定目录
2. 只检查新增或更新的笔记
3. 执行敏感内容过滤
4. 执行内容质量评分与候选筛选
5. 转写为 blog 可发布 MDX
6. 写入 `src/content/blog/` 下的目标路径
7. 自动 commit 到 blog repo
8. 触发 Vercel 自动部署

## 扫描策略：存量与增量分开处理

由于 Obsidian 仓库中通常已经有大量历史笔记，扫描策略必须区分为两种：

- **存量扫描（backfill）**：处理仓库里已经存在的大量旧笔记
- **增量扫描（incremental）**：处理后续新增或更新的笔记

这两者不能用同一种策略。

### 为什么不能直接全量深扫

如果对整个 vault 的所有笔记逐篇做深度分析，会有几个问题：

- 成本高
- 误判多
- 很多明显不适合公开的内容会浪费处理时间
- 大量碎片笔记会淹没有价值内容

因此推荐采用“先粗筛，再评分，再深筛”的三层扫描策略。

## 存量扫描（Backfill Scanner）

### 目标

不是一次性把所有旧笔记发布掉，而是先从整个仓库里筛出一个“高质量候选池”。

### 第一层：全库粗过滤

先对已有笔记做非常快的硬过滤，排除明显不该进入候选池的文件。

可直接过滤掉：

- Daily Notes
- Inbox
- Journal
- Life / Personal / Finance
- Career / Plan / Roadmap
- Meeting / Company Internal
- 纯任务清单
- 纯链接收藏
- 过短笔记
- 显式标记 private / internal 的笔记

这一层的目标是先把无关内容砍掉大半。

### 第二层：启发式评分

对剩下的笔记执行轻量评分，而不是直接转写发布。

评分信号可包含：

#### 加分项

- 位于 AI / Agents / LLM / Evals 等白名单目录
- 标题像文章题目，而不是随手记或日期
- 字数达到最低阈值
- 存在明确标题结构（如多个二级标题）
- 有结论、方法、步骤、复盘、对比等表达
- 有代码块、流程说明、结构化列表
- 有个人判断和总结，而不只是摘录

#### 减分项

- 文件名像日记、碎片、todo
- 任务列表占比过高
- 引用或摘抄占比过高
- 私人上下文过重
- 未来计划、路线图、内部讨论语义明显
- 太依赖背景，脱离原环境就无法公开阅读

### 第三层：候选池建立

根据评分，把存量笔记分成几档：

- `blocked`
- `low`
- `candidate`
- `strong-candidate`

后续只对 `candidate` 和 `strong-candidate` 做深度分析与转写。

这意味着：

- 不需要一开始处理整个仓库的所有笔记
- 可以先得到一批最值得发的内容候选
- 后续逐批处理，而不是一次性清库

## 增量扫描（Incremental Scanner）

### 目标

在系统跑起来后，只处理源 repo 里最近新增或更新的笔记。

### 检测方式

推荐基于 Git 变更做增量处理，例如：

- 最近一次扫描之后新增的文件
- 最近一次扫描之后修改过的文件
- 特定 commit 区间内发生变化的文件

这样可以避免每次重复扫整个仓库。

### 增量扫描流程

1. 找到最近一次处理后的 commit hash
2. 读取从该 commit 到当前最新 commit 的改动文件
3. 只对这些变更文件执行来源过滤、安全过滤、质量评分
4. 对高分候选执行转写
5. 更新导入状态文件

这样做的好处是：

- 扫描成本可控
- 更新速度快
- 更适合定时自动运行

## 候选池机制

推荐建立一个“候选池”而不是直接从源仓库一步到发布。

### 候选池的作用

- 让存量内容先沉淀为可处理列表
- 让系统知道哪些内容值得继续分析
- 让后续自动发布建立在候选清单之上，而不是全仓乱扫

### 候选池建议记录的信息

每条候选项可包含：

- 源笔记路径
- 标题
- 最后修改时间
- 最后处理 commit
- 质量评分
- 风险等级
- 分类结果（blocked / low / candidate / strong-candidate）
- 是否已导入
- 对应 blog slug（如果已有）

### 候选池文件建议

可以使用例如：

- `data/obsidian-candidates.json`

用于保存当前候选池结果。

## 评分机制建议

### 基本原则

评分的目的不是“判定绝对好坏”，而是排序。

也就是说，评分系统只需要回答一个问题：

> 在大量笔记里，哪些最值得优先转成 blog 内容？

### 推荐评分构成

可把评分拆成四部分：

- **主题匹配分**：是否符合 AI / agents / workflows / evals / product 方向
- **结构完整分**：是否具备标题层级、段落、列表、结论
- **内容价值分**：是否包含经验、方法、判断、案例
- **风险扣分**：是否带有敏感、私人、内部信息特征

最终结果不一定要是非常复杂的模型，简单的规则分就足够支撑 v1。

## 为什么这个策略适合“懒人自动化”

因为它解决的不是“每次你手动找一篇发”，而是：

- 第一次自动从大仓库里筛出最值得发的一批内容
- 之后只自动看新增和更新内容
- 让系统自己维护候选池和导入状态

这样你不需要每次手动翻整个 vault，只在少数例外场景下介入。

## 推荐运行模式

### 模式选择

推荐从 **半自动发布，自动提交** 起步，而不是一开始就“全自动公开”。

v1 推荐模式：

- 自动抓取
- 自动筛选
- 自动生成
- 自动提交到 blog repo 的 `drafts` 路径或 `imported/` 分类
- 达到高置信条件时再进入正式公开路径

### 为什么不建议一上来就全自动直发

因为误发一次敏感信息的成本远高于少发几篇文章。

因此建议采用：

- 默认保守
- 高风险内容宁可拦截
- 高质量低风险内容才自动进公开目录

## 内容进入 blog 的规则

内容必须同时满足“来源规则”“安全规则”“质量规则”。

### 1. 来源规则

只允许来自明确范围内的笔记，例如：

- `AI/`
- `Agents/`
- `LLM/`
- `Evals/`
- `Products/AI/`

默认不处理：

- Daily Notes
- Inbox
- Journal
- Life
- Finance
- Personal
- Career Planning
- Company Internal

也就是说，目录本身就是第一层安全网。

### 2. 安全规则

命中以下任一类内容时，禁止自动公开：

- 个人隐私
- 家庭/关系/联系方式
- 地址、手机号、邮箱、账号
- 未来计划、路线图、未公开目标
- 公司敏感信息
- 客户名称、项目代号、会议内容
- 内部文档引用
- 未脱敏的截图、表格、数据、对话
- 任何明确标记为 private / confidential / internal 的内容

### 3. 质量规则

即使内容安全，也不代表适合 blog。

自动筛选时优先选择：

- 有清晰主题
- 可提炼为一个明确观点
- 有方法、经验、流程、案例之一
- 不是流水账
- 不是纯收藏链接
- 不是只写情绪或想法碎片

适合自动发布的典型内容：

- 某个 agent workflow 的复盘
- 某种 prompt 或 eval 方法的经验总结
- 一个 AI 产品设计判断
- 一次工具编排的实践心得
- 某个模型行为的稳定观察

## 建议的发布分级

为降低风险，建议把候选内容分成三档：

### A. Blocked

直接禁止发布。

触发条件：

- 命中敏感信息规则
- 位于黑名单目录
- 显式标记为 private

### B. Draft

允许转成 blog 草稿，但不进入正式公开路径。

触发条件：

- 内容安全，但质量不稳定
- 结构散，需要重写
- 主题有价值，但表达还不适合公开

### C. Auto Publish

允许自动发布。

触发条件：

- 来源目录正确
- 未命中敏感风险
- 内容结构完整
- 质量评分达到阈值
- 主题明确且适配站点定位

## 敏感信息防护设计

推荐采用“三层防护”。

### 第一层：目录过滤

只扫描白名单目录。

### 第二层：元数据过滤

如果笔记存在以下属性，直接拦截：

- `private: true`
- `publish: false`
- `sensitivity: high`
- `status: internal`

即使你不想手动维护很多字段，至少也建议保留：

- `publish: false`
- `private: true`

这两个字段可以作为强制否决开关。

### 第三层：文本规则过滤

脚本扫描正文内容，命中下列特征时自动拦截：

- 邮箱、手机号、身份证样式、地址样式
- 公司名、客户名、项目代号字典
- 明显的 roadmap / plan / Q3 / OKR / hiring / budget 语义
- 会议纪要、聊天记录、决策过程原文
- 图片链接、附件、截图引用

未来可以再加一层 LLM 审核，但 v1 不建议完全依赖模型判断，应该以规则优先。

## 内容转写策略

Obsidian 笔记通常不直接等于 blog 文章，因此需要一层“转写”。

### 输入

源笔记 Markdown。

### 输出

符合 Astro content collection 的 MDX，例如：

- `src/content/blog/agents/<slug>/index.mdx`
- `src/content/blog/llm-workflows/<slug>/index.mdx`

### 转写动作

- 提取标题
- 生成摘要
- 清理 Obsidian 特有语法
- 去掉内部链接、任务列表、私密引用
- 重排为 blog 更适合的结构
- 保留必要代码块
- 必要时生成更通顺的小标题

### 建议的 blog frontmatter

```yaml
---
title: "..."
description: "..."
pubDate: "2026-03-25"
updatedDate: "2026-03-25"
source: "obsidian-import"
sourcePath: "AI/Agents/..."
importedAt: "2026-03-25T10:00:00Z"
---
```

说明：

- `sourcePath` 便于回溯原笔记
- `source` 便于后续筛选“自动导入内容”
- `importedAt` 有助于调试和审计

## 去重与更新策略

需要避免“每次都重复导入一篇老笔记”。

推荐机制：

- 以原笔记路径 + 标题 + 内容摘要哈希作为主识别依据
- 建立一份导入状态文件，例如：
  - `data/obsidian-import-state.json`

状态文件记录：

- 源笔记路径
- 最后处理 commit hash
- 最后导入时间
- 当前 blog slug
- 发布状态

这样可以实现：

- 新笔记自动导入
- 已导入笔记只在内容有明显变化时更新
- 被删除或撤回的源笔记可触发 blog 侧下线策略

## 自动化运行方式

### 推荐：GitHub Actions

在 blog repo 中配置定时任务：

- 每天 1 次
- 或每 6 小时 1 次

动作流程：

1. checkout blog repo
2. 拉取 Obsidian source repo
3. 运行导入脚本
4. 如果有变更则 commit
5. push 到 `main` 或指定分支
6. Vercel 自动部署

### 为什么优先 GitHub Actions

- 稳定
- 持续运行成本低
- 与 GitHub repo 源天然兼容
- 后续可接通知、PR、审计

## 权限与安全

如果 Obsidian repo 是公开仓库：

- 直接可读
- 配置最简单

如果 Obsidian repo 是私有仓库：

- 需要一次性提供只读访问能力
- 推荐使用 GitHub token 或 deploy key

当前已知源仓库：`Taz-dingo/obsidian-vault`

建议原则：

- 自动化任务只给源仓库读权限
- 对 blog repo 才给写权限
- 不要复用高权限个人 token

## 建议的目录与文件结构

在当前 blog repo 中增加：

- `scripts/obsidian-import/`
  - 导入主程序
  - Markdown 清洗器
  - 敏感规则检查器
  - 质量评分器
- `config/obsidian-import.config.json`
  - 白名单目录
  - 黑名单目录
  - 敏感关键词
  - 公司/客户黑名单词典
  - 自动发布阈值
- `data/obsidian-import-state.json`
  - 导入状态缓存
- `.github/workflows/obsidian-import.yml`
  - 定时任务

## 分阶段落地建议

### Phase A — 安全起步

目标：先把系统跑通，但绝不自动公开高风险内容。

包含：

- 从 GitHub repo 读取指定目录
- 做目录与文本双重过滤
- 生成 blog draft
- 只写入 `src/content/blog/imported/`
- 自动 commit

### Phase B — 自动优化

目标：减少人工干预。

包含：

- 增加质量评分
- 增加增量更新逻辑
- 自动生成摘要与 slug
- 自动更新已导入文章

### Phase C — 高置信自动发布

目标：让低风险高质量笔记自动公开。

包含：

- 建立“可自动公开”的规则阈值
- 针对特定目录直接进入正式路径
- 针对中风险内容只进草稿目录

## v1 明确不做的事

为了防止系统过重，v1 不建议一开始就做：

- 复杂 CMS
- 完全依赖 LLM 审核敏感性
- 图片自动处理与封面生成
- 多渠道分发
- 评论、推荐、阅读时间联动
- 复杂双向同步

## 推荐的默认决策

如果现在就开始实现，推荐默认按以下决策推进：

- 主数据源：Obsidian GitHub repo
- 备用调试源：本地 vault
- 不使用 OSS 作为主源
- 调度方式：GitHub Actions
- 默认模式：自动抓取 + 自动筛选 + 自动提交
- 发布策略：高风险拦截，低风险内容优先草稿化
- 正式公开：等规则稳定后再放开自动发布

## 当前最值得先做的第一版

最小可用版本建议只做这 5 件事：

1. 接入 Obsidian GitHub repo 读取能力
2. 只扫描 1 到 2 个白名单目录
3. 建立敏感词和黑名单目录过滤
4. 自动生成 blog draft 到固定路径
5. 自动 commit 到当前 blog repo

这个版本跑通之后，再加：

- 质量评分
- 自动摘要优化
- 正式发布阈值
- 删除/更新同步策略

## 需要你后续补充的信息

后续实现时需要补齐这些配置：

- Obsidian 源 repo 地址
- 源 repo 是否私有
- 哪些目录允许被扫描
- 哪些目录必须永久排除
- 公司/客户/项目黑名单词典
- 是否允许直接写 `main`

## 结论

这套方案的核心思想是：

- 用 GitHub repo 解决“自动持续读取”
- 用规则系统解决“敏感内容风控”
- 用内容转写解决“笔记不等于文章”
- 用 GitHub Actions + Vercel 解决“自动发布”

它能把 blog 维护方式从“手工挑选 + 手工复制 + 手工发布”转成：

- 自动发现
- 自动筛选
- 自动生成
- 自动部署

同时保留足够强的安全边界，避免把不该公开的内容带到线上。
